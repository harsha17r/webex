# Webex Design — Claude Code Rules

## 🔴 MANDATORY: Development Log

**Every ~30 minutes of active coding**, you MUST write a dev log entry. This is a time-based log, not a per-change log.

### Timing Rules
- **Track the clock.** After your first code change in a session, note the time. Every ~30 minutes of work after that, write a log entry.
- **At session end**, always write a final entry regardless of timing — never leave without logging.
- **If the user asks you to stop or you're done early**, write the entry before wrapping up.
- **Don't log every tiny change** — batch your work into one 150–200 word summary per 30-minute window.

### How It Works

1. **Log file location:** `devlog/LOG.md`
2. **Create the file and folder** if they don't exist yet.
3. **Every entry** must follow this exact format:

```markdown
---

### [YYYY-MM-DD HH:MM] — <Short Title>

**Status:** 🟢 Done | 🟡 In Progress | 🔴 Blocked

**What changed:**
<150–200 word summary of what was built, modified, or fixed. Be specific — mention file names, component names, routes, and logic changes.>

**Files touched:**
- `path/to/file1.tsx`
- `path/to/file2.css`

**Next up:**
<What should be worked on next. Be specific enough that someone returning after a break knows exactly where to pick up.>
```

4. **New entries go at the TOP** of the file (newest first), below the header.
5. The log header should look like this (create once):

```markdown
# 📒 Webex Design — Dev Log

> Auto-generated by Claude Code. Newest entries first.
> Read this file to catch up on where the project stands.

```

### Rules
- **Log every ~30 minutes**, not every micro-change. Batch your work into one clear summary.
- **Be specific.** Don't write "updated styles" — write "added dark mode toggle to `Navbar.tsx`, updated Tailwind classes for `bg-gray-900` in meeting cards."
- **Always include "Next up"** so the user (or you in a future session) knows what to do next.
- **Always log before ending a session**, even if 30 minutes haven't passed yet.
- **If you're resuming work**, read `devlog/LOG.md` FIRST before doing anything to understand the current state.

## Code Style & Conventions

- Use functional components with hooks (no class components)
- Tailwind CSS for all styling — no separate CSS files unless absolutely necessary
- Use React Router for navigation (`react-router-dom`)
- Keep components small and composable
- Name files in PascalCase for components (`MeetingCard.tsx`), camelCase for utils (`formatDate.ts`)
- Prefer named exports over default exports for components

---

## Session Start Checklist

When starting a new session:
1. ✅ Read `devlog/LOG.md` to catch up
2. ✅ Check the latest "Next up" section
3. ✅ Confirm the plan with the user before coding
4. ✅ After finishing, write the log entry
